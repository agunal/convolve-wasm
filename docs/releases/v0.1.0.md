# @takana-labs/convolve-wasm v0.1.0

**Status:** Draft release notes. Do not publish, tag, or create a GitHub release until final merged-commit verification, exact-tarball approval, and the one-time first-publication controls are complete.

## Overview

`@takana-labs/convolve-wasm` is a browser-side stereo audio convolution library backed by a Rust/WebAssembly DSP core and a dedicated module worker. Source audio and rendered output remain on the client.

## Highlights

- Public `CONVOLVE()` API for `.wav` and browser-decodable `.m4a` inputs.
- Full wet-only stereo linear convolution at 48 kHz.
- Optional beat detection from either input and equal-power alternating left/right panning.
- Optional exact sample-time reverse append with a complementary midpoint crossfade.
- Attenuation-only estimated true-peak normalization.
- Stereo 48 kHz signed 24-bit PCM WAV output.
- Dedicated module worker with transferable buffers and progress events.
- Exact 256 MiB conservative WASM allocation guard.
- Stable public error codes.
- Package-consumer tests proving that worker and standalone WASM assets remain local and bundle correctly.
- No bundled `@ffmpeg/core` dependency.

## Public API

```ts
import { CONVOLVE } from "@takana-labs/convolve-wasm";

const result = await CONVOLVE(
  { a: fileA, b: fileB },
  true,
  {
    beatPan: "a",
    panTransitionMs: 20,
    reverseCrossfadeMs: 5,
    targetDbtp: -1,
  },
);
```

```ts
CONVOLVE(
  audio: { a: File; b: File },
  appendReverse?: boolean,
  options?: ConvolveOptions,
): Promise<ConvolveResult>
```

## Browser and codec boundary

WAV is the automated portability baseline. M4A decoding uses the browser and operating system media stack through `decodeAudioData()` and may return `DECODE_FAILED` when the required codec is unavailable.

Automated WAV tests passed in Chrome through `wasm-pack`, Playwright Chromium, and Playwright WebKit. The release HE-AAC matrix also passed against the unchanged production application in branded Chrome 149 on Windows Server 2025, branded Edge 149 on Windows Server 2025, and system Safari 26.4 on macOS 26.4. Every browser passed plain convolution, beat pan from A with reverse append, Blob playback, download-byte capture, browser-specific output-frame formulas, stereo 48 kHz PCM24 validation, finite non-silent peak checks, clipping checks, and page-error checks.

See [browser and codec support](../browser-support.md) and [the hosted release-browser evidence report](../testing/2026-07-15-hosted-release-browser-matrix.md) for exact versions, fixture hashes, output hashes, formulas, and artifact identities.

## Known limitations

- M4A support is browser/OS codec dependent rather than deterministic.
- Firefox M4A behavior is best effort and platform dependent.
- Processing is single-threaded WASM in a dedicated worker.
- Beat panning intentionally collapses the convolved stereo pair to one mono spatial object.
- v0.1.0 does not add dither.
- No server fallback or bundled FFmpeg decoder is included.

## First-publication bootstrap

A registry probe on 2026-07-15 confirmed that neither `@takana-labs/convolve-wasm` nor version `0.1.0` exists on npm. npm trusted publishing is configured from an existing package's settings, and staged publishing cannot create a brand-new package. Therefore the first release requires one tightly controlled granular-token publish from the protected GitHub workflow.

The first publish still occurs on a GitHub-hosted runner with provenance and uses only the exact previously inspected tarball. The short-lived bootstrap token must be revoked immediately afterward. Then npm trusted publishing must be configured for `agunal/convolve-wasm`, workflow `publish.yml`, environment `npm`, and the package must be set to disallow traditional tokens.

## Verification required before release

The final release must be produced from the exact commit created by merging the final release pull request. From that commit:

1. Complete clean-room Rust, WASM, TypeScript, package-consumer, build, Chromium, WebKit, and package checks.
2. Pack once and retain the exact `.tgz`.
3. Inspect the exact tarball, including its manifest, worker, declarations, WASM asset, and absence of `@ffmpeg/core`.
4. Record the source commit, candidate workflow run, tarball filename, metadata, and SHA-256.
5. Obtain explicit authorization naming those exact values and the one-time bootstrap mode.
6. Create a protected GitHub `npm` environment and add the short-lived `NPM_BOOTSTRAP_TOKEN` only as an environment secret.
7. Tag the authorized commit and publish the already inspected tarball with provenance without rebuilding or repacking.
8. Revoke the token, delete the secret, configure npm trusted publishing, and disallow traditional package tokens.
